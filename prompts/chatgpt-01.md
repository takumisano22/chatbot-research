QA50~100のデータセットを、1ワークフローとしてバッチ実行する用の実験専用プロジェクトを作成しています。添付zipが現在の実装です。下記の要件で修正、実装するためにcursorに投げるプロンプトを作成してください。 

#要件 
-実行は、github Actionsで、self-host runnerを遠隔においたデスクトップで動かせるようにしたいです。 
-データセット1セットの実行のたびに、vectorDBのコレクションは空にしてから、再度pdfを取り込んだ上で、推論を実行するようにしてください。 
-取り込みpdfは、backendの外側にingest_documentディレクトリを作って、中にいくつかのid付きディレクトリを作成できる＆実行時にどれを読むか指定できる設計をしてください。 
-.envとresearch_pairなどでファイルの責務を分けて、.env側にAPI設定やホストやポートの実態の記載を、research_pair側には、そのワークフローで使うロジックやLLMの選択を簡潔に書けるようにしてください。 
-実行当たりで変更するのは、基本的にresearch_pairの指定のみとし、変更したらpush（orなんらかのコマンド）すると実行されるイメージで構築してください。 
-計測はlangfuse+RAGASで行うものとし、最終的な成果は、「選択research_pair（×項目数分）,入力,出力,RAG検索結果ドキュメント×topK個,distance×topK個,RAGAS観測結果」をcsvで取得してください。吟味した上なら、必要に応じて項目は追加しても構いません。 
-csvは実行しているデスクPCのローカルに保存で大丈夫です。 
-RAGASの使用はON,OFFを切り替えられるものとし、OFFの際はNoneなどで成果csvは埋めて列数はそろえてください。 
-research_pairの項目は、現時点で「推論LLM、埋め込みLLM、検索ロジック、チャンキングロジック、再ランキングロジック、トークナイザーロジック、トップK、使用QAデータセット」の8つです。 
-topKはresearch_pairのテキスト変更で直接指定、使用QAデータセットは事前にbackendの外に置いておいてファイル名指定、推論LLMと埋め込みLLMは現時点の実装を切り替えできれば十分とします。 
-「検索ロジック、チャンキングロジック、再ランキングロジック、トークナイザーロジック」の4つについては、各ロジックごとにディレクトリをつくり、その中に~_LOGIC-01.py,~_LOGIC-02.pyのように番号指定したファイルを並列でそれぞれおいて、research_pairでどれを使用するか決められるようにしてください。 
-使用するQAデータセットは、json形式などで記述し、事前に特定のディレクトリに入れておくものとします。 

#注意点 
-KISS原則を意識して、不要な実装は行わないでください。 
-バッチ実行を前提として必要十分の機能を保ちながら、なるべく軽い負荷で実行できるようにしてください。 
-機能を満たしていれば、不要な実装は削除して構いません。 
-設計を十分に行ったうえで、実装を開始してください。 

#運用方針 
下記の運用方針をよく理解したうえで、プロンプトの設計を行ってください。このリポジトリ内で実装するのは、Dockerにあたる部分だけであることに注意してください。 

-GitHub Actions 
・実験開始のトリガー 
・ジョブ実行の制御 
・ログと成果物の回収 

-self-hosted runner（Windows デスクトップ上） 
GitHub Actions の job を受け取って実行する主体 

-Docker QA バッチ処理の固定実行環境 

-Windowsデスクトップ ローカルホスト上の LLM サーバー 
推論リクエストの実行先 

-Tailscale + ノートPCからデスクPCへリモートデスクトップ接続 
設定変更、障害調査、GUI確認時のみ使用
