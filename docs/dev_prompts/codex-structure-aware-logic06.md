# チャンキングロジック&検索最適化プロンプト

## 役割と指示
- あなたは、チャンク分割のロジックに詳しく、作業者のコード実装をサポートするエージェントです。
- chunking_logic_06.pyに現在の実装をコピーしたので、このファイルを編集し、下記の内容を実装してください。


### 設計
既存のコードでは、マークダウン化した文書を直接ベクトル化して、推論LLMにも同じテキストを渡しています。それを下記の設計に変更したいです。
- 一つのチャンク内でベクトル化する文書を最大二種類に分岐して両方のデータを保持するようにしてください。
  1. 既存のもの（上層ノードが全てついたもの）。
  2. 子チャンクもしくは孫チャンクの場合、「親ノード（"##"）部分と文書ノード（"#"）」を"除去"したもの。ただし、"###"で始まる子ノードは下層チャンクでも残すこと。
     - 親チャンクは、2.のデータを持たなくていいです。
- さらに、ベクトル化する文書と、推論LLMに渡す文書のデータを分けて保持してください。
  - ベクトル化する文書はマークダウンを外したうえで、ベクトル化してください。。


### 要件
- 上記の設計で、一つのチャンクは最大3種類のテキストと、最大2種類のベクトルデータを持つはずです。これらをchromaDBに保存する想定で、設計を行ってください。
  - 必要に応じて、chunking_logic_06.py以外のベクトルDBにかかわるコードも修正して構いません。ただし、他の既存ロジックを使っても動くように、互換性を保ってください。
    - ロジックによって、保存するメタデータ・ベクトルデータの数が変わることを許容できるコード記述が望ましいです。
  - metadataの作り方としては、それぞれが自明にその文書のなんのmetadataとして保持されるかわかる命名規則としてください。
    - 検索後にLLMに渡す文書は一意なので、それがdocumentとして保持されるなら、metadataに二重に含む必要はありません。chromaDBのデータ保持と検索ロジックを考慮して適切な設計を行ってください。
- 検索ロジックについても、簡単に実装してください。backend/app/rag/logic/search/に、search_logic_03.pyとして、互換性を保つように実装してください。
  - 検索時に、チャンクの持つベクトルとの距離をそれぞれのベクトルで計算して、どちらもTopKを算出する際に選択肢にいれるやり方で実装してください。
    - TopKの算出後に、同チャンクの候補が複数残った場合も、現時点ではドキュメントを複数個渡してよいです。
  - 互換性を持たせる設計をしてください。
